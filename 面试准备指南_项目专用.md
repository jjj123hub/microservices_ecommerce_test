# 微服务在线商城质量保障项目 - 面试准备指南

## 项目概述
这是一个基于微服务架构的电商系统质量保障项目，包含 UI 自动化测试、API 契约测试、性能测试和 Redis 热点 Key 优化等。

## 技术栈详解

| 技术 | 用途 | 核心原理 |
|------|------|----------|
| **Playwright** | UI 自动化 | 微软开发，多浏览器支持，自动等待 |
| **pytest** | 测试框架 | 同上 |
| **Pact** | 契约测试 | 消费者驱动契约，确保服务间API兼容 |
| **k6** | 性能测试 | Go 语言开发，高性能负载测试 |
| **Jira** | 缺陷跟踪 | 同上 |
| **MySQL** | 数据库 | 同上 |
| **Linux** | 环境 | 同上 |
| **Redis** | 缓存优化 | 内存数据库，热点 Key 优化 |

## Playwright 深度解析

### 核心优势（对比 Selenium）

| 特性 | Playwright | Selenium |
|------|-----------|---------|
| **自动等待** | ✅ 自动等待元素可交互 | ❌ 需要手动等待 |
| **多浏览器** | ✅ Chromium、Firefox、WebKit | ✅ 需要配置 |
| **网络拦截** | ✅ 原生支持 Mock API | ⚠️ 需要第三方库 |
| **跨平台** | ✅ Windows、Mac、Linux | ✅ 同上 |
| **追踪功能** | ✅ 完整执行录制 | ⚠️ 需要额外工具 |

### 自动等待机制
Playwright 会自动等待元素满足以下条件才执行操作：
- 元素已附加到 DOM
- 元素可见
- 元素可交互（非禁用、非只读）
- 元素不在动画中
- 元素不在视口外（会自动滚动）

### 面试要点
- **Playwright 为什么比 Selenium 快？**
  - 直接使用 DevTools Protocol（Chrome）
  - 多浏览器上下文并行
  - 智能等待减少不必要的 sleep

- **什么是 Playwright Trace？**
  - 完整记录测试执行过程
  - 包含 DOM 快照、网络请求、控制台日志
  - 便于调试 flaky tests

## 契约测试 (Contract Testing)

### 什么是契约测试？
确保服务消费者和提供者之间的 API 契约保持一致，防止集成问题。

### Pact 工作流程

```
┌─────────────────┐         ┌─────────────────┐
│   消费者端      │         │   提供者端      │
│  (Consumer)     │         │  (Provider)     │
└────────┬────────┘         └────────┬────────┘
         │                             │
         │  1. 编写测试，定义期望       │
         │                             │
         │  2. 生成 Pact 文件         │
         │  ─────────────────────────> │
         │   (JSON 格式)               │
         │                             │
         │                             │  3. 验证是否满足契约
         │  <───────────────────────── │
         │   验证结果                   │
         │                             │
```

### 核心概念
- **消费者**: 调用 API 的服务
- **提供者**: 提供 API 的服务
- **Pact 文件**: 描述 API 契约的 JSON 文件
- **提供者状态**: 测试前的数据准备

### 面试要点
- **契约测试 vs 集成测试？**
  - 契约测试: 快速、独立、可在本地运行
  - 集成测试: 真实、全面、需要部署环境
  - 两者互补，不是替代关系

- **Pact 的优势？**
  - 提前发现 API 变更问题
  - 减少集成测试成本
  - 支持服务独立部署

## 性能测试与优化

### k6 性能测试工具

**特点**:
- 基于 Go 语言，性能优异
- 使用 JavaScript 编写测试脚本
- 支持多种协议（HTTP、WebSocket、gRPC）
- 丰富的指标和阈值设置

**核心概念**:
- **VU (Virtual User)**: 虚拟用户
- **Iteration**: 测试脚本执行一次
- **Thresholds**: 性能阈值（如 P95 < 500ms）
- **Stages**: 负载阶段（逐步增加用户）

### Redis 热点 Key 优化

**什么是热点 Key？**
- 访问频率极高的 Key
- 可能导致 Redis 节点负载不均
- 严重时可能导致缓存雪崩

**热点 Key 检测方法**:
1. **TTL 分析**: 查看 Key 的过期时间
2. **访问频率**: 统计 Key 的访问次数
3. **内存占用**: 大 Key 也可能成为热点

**优化策略**:

| 策略 | 原理 | 适用场景 |
|------|------|---------|
| **本地缓存** | 应用层 L1 Cache，减少 Redis 访问 | 读多写少，数据允许短暂不一致 |
| **Key 分片** | 将热点 Key 分散到多个 Key | 计数器类场景 |
| **读写分离** | 主从复制，读请求走从库 | 读远多于写 |
| **限流熔断** | 保护后端系统不被压垮 | 高并发场景 |

**面试要点**:
- **P99 响应时间是什么？**
  - 99% 的请求响应时间低于该值
  - 比平均值更能反映系统性能
  - 项目优化效果: 1.2s → 320ms

- **如何定位 Redis 热点 Key？**
  - 使用 `redis-cli --bigkeys`
  - 监控 `info stats` 中的 keyspace_hits
  - 应用层埋点统计

### Q2: Playwright 和 Selenium 相比有什么优势？

**参考答案**:
> Playwright 相比 Selenium 有几个明显优势：
> 
> 1. **自动等待**: Playwright 会自动等待元素可交互才执行操作，大大减少了 flaky tests
> 2. **多浏览器支持**: 同一套代码可以在 Chromium、Firefox、WebKit 上运行
> 3. **网络拦截**: 原生支持 Mock API，便于测试各种场景
> 4. **追踪功能**: 可以录制完整的测试执行过程，包含 DOM 快照、网络请求等，调试非常方便
> 
> 我们项目用 Playwright 后，回归测试效率提升了 80%。

---

### Q3: 什么是契约测试？为什么需要契约测试？

**参考答案**:
> 契约测试是确保服务消费者和提供者之间的 API 契约保持一致的测试方法。
> 
> **为什么需要**:
> - 微服务架构下，服务间依赖复杂，API 变更容易导致集成问题
> - 集成测试需要部署所有服务，成本高、速度慢
> - 契约测试可以在本地快速运行，提前发现问题
> 
> **Pact 工作流程**:
> 1. 消费者端编写测试，定义期望的请求和响应
> 2. 生成 Pact 文件（JSON 格式）
> 3. 提供者端验证是否满足契约
> 
> 我们项目引入 Pact 后，联调问题减少了 70%。

---

### Q4: 如何定位和优化 Redis 热点 Key？

**参考答案**:
> **热点 Key 的定位**:
> 1. 使用 `redis-cli --bigkeys` 查找大 Key
> 2. 分析 Key 的 TTL 和访问频率
> 3. 应用层埋点统计访问次数
> 
> **优化策略**:
> 1. **本地缓存**: 应用层加 L1 Cache，比如 Guava Cache，减少 Redis 访问
> 2. **Key 分片**: 将热点 Key 分散到多个 Key，比如 `hotkey_1`, `hotkey_2`...
> 3. **读写分离**: 主从复制，读请求走从库
> 4. **限流熔断**: 保护后端系统
> 
> 我们项目通过优化 Redis 热点 Key，接口响应时间 P99 从 1.2s 降到了 320ms。

---

### Q5: 性能测试中 P95、P99 是什么意思？

**参考答案**:
> **P95**: 95% 的请求响应时间低于该值
> **P99**: 99% 的请求响应时间低于该值
> 
> **为什么重要**:
> - 平均值容易被异常值影响
> - P95/P99 更能反映系统的真实性能体验
> - 对于电商系统，P99 响应时间直接影响用户转化率
> 
> 我们项目优化前 P99 是 1.2s，优化后降到了 320ms。

---

### Q6: 微服务架构下的测试策略是什么？

**参考答案**:
> 遵循测试金字塔：
> 
> 1. **单元测试** (60%): 验证单个函数逻辑，快速反馈
> 2. **集成测试** (30%): 验证服务间交互，用 Pact 做契约测试
> 3. **E2E 测试** (10%): 覆盖核心业务流程，用 Playwright
> 4. **性能测试**: k6 做压测，关注并发和响应时间
> 
> **测试关注点**:
> - 服务可用性
> - 数据一致性
> - 故障隔离
> - 性能指标
